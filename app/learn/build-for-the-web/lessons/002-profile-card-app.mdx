### What `home.tsx` file represents

- The `home.tsx` file represents a **page** in your application.
- In React Router v7 (Framework Mode), a file named `home.tsx` inside `app/routes`
  automatically becomes the **home page** of your app.

### Line-by-line explanation

#### 1Ô∏è‚É£ `export default`

```ts
export default function Home() {
```

- This makes the function available to the rest of the app.
- React Router expects each route file to export **one default component**.
- That exported function is what gets rendered on the page.

- Think of it as:

  > This is the main thing this file provides.

<hr className='my-8' />

#### 2Ô∏è‚É£ `function Home()`

```ts
function Home() {
```

- This defines a JavaScript function named `Home`.
- In React, a function like this is called a **component**.
- A component's job is simple:
  > Return UI
- The name `Home` matches the route:
  - `home.tsx` ‚Üí `Home` component ‚Üí home page

#### 3Ô∏è‚É£ `return (...)`

```ts
return (
```

- Whatever is returned here becomes **visible in the browser**.
- React reads this return value and turns it into real HTML.
- If a component returns nothing, nothing appears on the page.

#### 4Ô∏è‚É£ JSX: HTML-like syntax inside JavaScript

```tsx
<h1 className="text-4xl font-bold text-red-600 m-4">
```

- This looks like HTML, but it's actually JSX:
  - JSX lets you write HTML-like elements inside JavaScript.
  - React converts JSX into real DOM elements.
- Important detail:
  - In JSX, we use `className`, not `class`.
  - This is because `class` is a reserved word in JavaScript.

#### 5Ô∏è‚É£ The `<h1>` element

```tsx
<h1>Home page</h1>
```

- `<h1>` is a standard HTML heading.
- It represents the **main heading** of the page.
- Screen readers and search engines treat it as the page's title.

#### 6Ô∏è‚É£ Tailwind CSS classes (styling)

```tsx
className = 'text-4xl font-bold text-red-600 m-4'
```

- These classes come from Tailwind CSS, a utility-first CSS framework.
- Here's what each class does:
  - `text-4xl` ‚Üí makes the text large.
  - `font-bold` ‚Üí makes the text bold.
  - `text-red-600` ‚Üí applies a red color.
  - `m-4` ‚Üí adds margin around the heading.

- These styles:
  - Apply only to this element
  - Do not affect the rest of the page

    1. The function returns JSX

## Step 5: Add interaction and finishing touches ‚ú®

- At this point, the Profile Card:

### 5a: Add hover feedback to the action buttons

- Buttons should **respond** when the user interacts with them.
- We do this with:
  - Cursor feedback
  - Hover states
  - Subtle transitions
- No JavaScript.
- No state.
- No click logic.
- **Update the actions section**

```tsx
{/* Actions */}
<div className='mt-6 flex gap-3'>
	<button
		type='button'
		className='flex-1 cursor-pointer rounded-lg border px-4 py-2 text-sm transition hover:bg-slate-100'
	>
		Message
	</button>
	<button
		type='button'
		className='flex-1 rounded-lg bg-blue-600 px-4 py-2 text-sm text-white transition hover:bg-blue-700'
	>
		Follow
	</button>
</div>
```

- **Why the buttons are styled differently**
  - Not all buttons are equal.
  - One action is **primary**.
  - One action is **secondary**.
- This difference is intentional.

- **Class-by-class breakdown üß©**
  - **Message button (secondary action)**
    - `flex-1`
      ‚Üí keeps both buttons the same width.
    - `cursor-pointer`
      ‚Üí explicitly signals ‚Äúthis is clickable.‚Äù
    - `border`
      ‚Üí keeps the button visible but understated.
    - `hover:bg-slate-100`
      ‚Üí provides gentle feedback without stealing attention.
    - `transition`
      ‚Üí makes the hover feel smooth, not abrupt.

- This button says:

  > I'm available, but I'm not the main focus.

- **Follow button (primary action)**

```tsx
className='flex-1 rounded-lg bg-blue-600 px-4 py-2 text-sm text-white transition hover:bg-blue-700'
```

- `bg-blue-600`
  ‚Üí establishes visual priority.
- `text-white`
  ‚Üí ensures contrast and readability.
- `hover:bg-blue-700`
  ‚Üí reinforces intent on interaction.
- `transition`
  ‚Üí keeps interaction smooth.

Notice:

- We don't need cursor-pointer here.
- Buttons already show a pointer cursor by default.
- The strong color already communicates interactivity.

- This button says:
  > This is the action you're most likely to take.

### Why we stop here

- In this step, we intentionally do **not** add:
  - Click handlers
  - State
  - Modals
  - Application logic
- That belongs in a future lesson.
- What mattered here was learning how to:
  - Signal interactivity
  - Establish action priority
  - Add feedback without behavior
- Not application behavior.

### Key takeaway üß†

- Interaction doesn't always mean JavaScript.
- Visual feedback can communicate intent.
- Primary and secondary actions should feel different.
- A UI can feel alive **before** it becomes dynamic.
