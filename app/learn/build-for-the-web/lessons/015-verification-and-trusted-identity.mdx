## Introduction 🧭

- In Lesson 14, the system learned how to control flow.
- The application can now:
  - Create a session
  - Redirect users
  - Protect routes
  - Destroy sessions

```text
Request → Session → userId → Decision → Response
```

- The system now knows:

  > What should happen after identity exists?

- That was a major step.
- But something is still missing.

#### The Problem

- Open:

```bash
app/routes/login.tsx
```

- Your login route currently looks like this:

```ts
export async function action({ request }: Route.ActionArgs) {
	return login(request, 'user-1')
}
```

- And your UI contains a single button:

```tsx
<button
	type='submit'
	name='userId'
	value='user-1'
>
	Log in as User 1
</button>
```

- Right now, login behaves like this:

```tsx
Click button → login(request, 'user-1') → Session created
```

- This means:

```tsx
Any login request → same identity
```

- No matter who submits the form:
  - The same user is assigned
  - The same session is created
  - The same identity is trusted
- The system does not check:
  - Who submitted the request
  - Whether a user exists
  - Whether credentials are valid
- Authentication happens immediately.
- Validation never happens.

#### Reality

```tsx
Login request → identity assumed
```

- The system creates a session.
- But it never asks:

> Who is actually logging in?

- There is no:
  - Email lookup
  - Password validation
  - User verification

#### What this means

- The application knows:

> How to create authentication

- But it does not know:

> Whether authentication should happen

- Identity exists.
- But identity is not trusted.

#### Reality

- A real system should behave like this:

```text
Email + Password → Verify User → Create Session
```

- Authentication should not assume identity.
- Authentication should validate identity.

#### The Real Question

- A real system must decide:

> Does this request belong to a real user?

- Not assumed
- Not hardcoded
- But verified on every login request

#### What This Lesson Introduces

- By the end of this lesson, you will:
  - Accept credentials
  - Read email and password
  - Find users from stored data
  - Verify credentials
  - Reject invalid authentication
  - Create sessions from real identity

> This lesson is about proving identity

## Step 4: Introduce Real User Lookup with Prisma 🔎

- Authentication requires stored identity.

#### The Problem

- Right now:

  ```text
  Credentials exist → no user lookup
  ```

- The system receives email and password.
- But it has nowhere to verify them.

#### Reality

```text
Email → Database → User → Validation
```

- A real system does not authenticate from memory.
- It authenticates from persistent data.

### Step 4a: Update the User Model

- Open:

  ```bash
  prisma/schema.prisma
  ```

- Your current `User` model looks like this:

  ```prisma
  model User {
    id   String @id @default(cuid())
    name String

    messages Message[]
    profile  Profile?
  }
  ```

- Right now:

  ```text
  User exists → no credentials stored
  ```

#### Update the model

- Replace the `User` model with:

  ```prisma
  model User {
    id       String @id @default(cuid())
    name     String
    email    String? @unique
    password String?

    messages Message[]
    profile  Profile?
  }
  ```

#### Why are the fields optional?

- Your database already contains users.
- Right now, existing rows do not contain:

  ```text
  email
  password
  ```

- If you add required fields immediately:

  ```text
  Existing rows → missing values → migration fails
  ```

- Making the fields optional allows Prisma to safely update the schema.

#### Reality

```text
Existing users → preserved
New auth fields → added safely
```

- Later, once all users contain credentials, these fields can become required.

#### What changed?

- Before:

  ```text
  User → display identity only
  ```

- After:

  ```text
  User → authentication identity
  ```

#### Why this matters

- Authentication requires lookup fields.

> Identity must be searchable

### Step 4b: Run Prisma Migration

- After updating the schema, run:

  ```prisma
  npx prisma migrate dev --name add-auth-fields
  ```

#### What this does

- Updates the database schema.
- Adds new columns.
- Synchronizes Prisma with SQLite.

#### What just happened?

- Before:

  ```text
  Database → no authentication fields
  ```

- After:

  ```text
  Database → supports credentials
  ```

### Step 5a: Update the Login Action

- Open:

```bash
app/routes/login.tsx
```

- Add the import:

```ts
import { getUserByEmail } from '~/data/user.server'
```

- Then update your action:

```tsx
export async function action({ request }: Route.ActionArgs) {
	const formData = await request.formData()

	const email = formData.get('email')
	const password = formData.get('password')

	if (typeof email !== 'string' || typeof password !== 'string') {
		return null
	}

	const user = await getUserByEmail(email)

	if (!user) {
		return null
	}

	if (user.password !== password) {
		return null
	}

	return login(request, user.id)
}
```

#### What this does

- Reads credentials
- Searches for a user
- Validates password
- Creates identity

```text
Request → Credentials → User → Validation → Session
```

#### What just happened?

- Before:

```text

```

- After:


## Step 6: Improve Invalid Login Behavior ⚠️

- There is another problem.
- Right now:

```text
Invalid credentials → return null
```

- The system rejects authentication.
- But the user does not know why.

#### Why this matters

- Authentication should communicate failure.

```text
Rejected identity → visible explanation
```

#### Reality

```text
Invalid credentials → error message
```

### Step 6a: Return an Error Object

- Inside your login action:
  - Replace:

    ```ts
    return null
    ```

  - With:

  ```ts
  return {
    error: 'Invalid email or password',
  }
  ```

### Step 6b: Read Action Data

- Add:

  ```ts
  import { useActionData } from 'react-router'
  ```

- Inside your component:

  ```ts
  const actionData = useActionData<typeof action>()
  ```

### Step 6c: Render the Error

- Below your form:

  ```tsx
  {actionData?.error ? (
    <p className='text-sm text-red-600'>{actionData.error}</p>
  ) : null}
  ```

#### What this does

- Failed authentication now becomes visible.

#### What just happened?

- Before:

  ```text
  Authentication failed → silence
  ```

- After:

  ```text
  Authentication failed → explanation
  ```

#### Why this matters

> Systems should explain rejected identity
