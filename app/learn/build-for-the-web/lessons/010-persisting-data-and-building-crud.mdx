## Introduction 🧭

- In Lesson 8, you made the system **correct**.
- In Lesson 9, you made the system **resilient**.
- But the system still has one fundamental limitation:
  > **Nothing survives a page reload.**
- All data currently lives in memory.
- When the process restarts:
  - All messages are gone.
  - All counts reset.
  - The system forgets everything.
- That means:
  - The system is interactive…
  - But not persistent.
  - And therefore not yet real.
- In this lesson, we cross that line.
- By the end of this lesson, you will:
  - Persist profile card messages across reloads.
  - Support **C**reate, **R**ead, **U**pdate, and **D**elete (CRUD) operations.
  - Turn the Profile Card into a real data-backed system.
- This is the moment where your app stops being a demo and starts being software.

## Mental Model 🧠

> **State disappears when the process stops.  
> Persistence survives restarts.**

- In-memory state dies with the process.
- Persistent state lives outside the process.
- A real system must remember beyond a single run.
- And:
  > **CRUD is not a feature. CRUD is the minimal interface required to
  > reason about persistent state.**
- Everything that persists must be:
  - **C**reated
  - **R**ead
  - **U**pdated
  - **D**eleted

```tsx
import { useState, useEffect } from 'react'

import { Form, useNavigation, isRouteErrorResponse } from 'react-router'

import type { Route } from './+types/home'

import { readStore, writeStore } from '~/data/store.server'
import { randomUUID } from 'crypto'

import cartoonAntelopeAvatar from '~/assets/Cartoon-Antelope-Avatar.png?url'

export type Message = {
	id: string
	text: string
	createdAt: number
}

export type Profile = {
	id: string
	isFollowing: boolean
}

type ValidationError = {
	ok: false
	error: string
}

type ValidationSuccess = {
	ok: true
	value: string
}

type ValidationResult = ValidationError | ValidationSuccess

function validateMessageText(text: unknown): ValidationResult {
	if (typeof text !== 'string') {
		return {
			ok: false,
			error: 'A message must be a string.',
		}
	}

	const trimmedText = text.trim()

	if (trimmedText.length === 0) {
		return {
			ok: false,
			error: 'A message must not be empty.',
		}
	}

	const minLength = 3
	const maxLength = 200

	if (trimmedText.length < minLength) {
		return {
			ok: false,
			error: `A message must be at least ${minLength} characters.`,
		}
	}

	if (trimmedText.length > maxLength) {
		return {
			ok: false,
			error: `A message must be at most ${maxLength} characters.`,
		}
	}

	return { ok: true, value: trimmedText }
}

export async function action({ request }: Route.ActionArgs) {
	// 👇 Uncomment to simulate a server-side failure
	// throw new Error('Simulated server failure')
	const formData = await request.formData()
	const intent = formData.get('intent')

	if (intent === 'create') {
		const messageText = formData.get('message')

		const validatedMessageText = validateMessageText(messageText)

		if (!validatedMessageText.ok) {
			return validatedMessageText
		}

		const store = await readStore()

		const newMessage = {
			id: randomUUID(),
			text: validatedMessageText.value,
			createdAt: Date.now(),
		}

		store.messages.push(newMessage)
		await writeStore(store)

		return { ok: true }
	}

	if (intent === 'update') {
		const id = formData.get('id')
		const messageText = formData.get('messageText')

		if (typeof id !== 'string') {
			return { ok: false, error: 'Invalid update payload.' }
		}

		const validatedMessageText = validateMessageText(messageText)

		if (!validatedMessageText.ok) {
			return validatedMessageText
		}

		const store = await readStore()
		const message = store.messages.find(m => m.id === id)

		if (!message) {
			return { ok: false, error: 'Message not found.' }
		}

		message.text = validatedMessageText.value
		await writeStore(store)

		return { ok: true }
	}

	if (intent === 'delete') {
		const id = formData.get('id')

		if (typeof id !== 'string') {
			return { ok: false, error: 'Invalid delete payload.' }
		}

		const store = await readStore()
		const initialLength = store.messages.length

		store.messages = store.messages.filter(m => m.id !== id)

		if (store.messages.length === initialLength) {
			return { ok: false, error: 'Message not found.' }
		}

		await writeStore(store)

		return { ok: true }
	}

	if (intent === 'toggle-follow') {
		const store = await readStore()

		store.profile.isFollowing = !store.profile.isFollowing

		await writeStore(store)

		return { ok: true }
	}

	return { ok: false, error: 'Unsupported intent' }
}

export async function loader() {
	const store = await readStore()
	return store
}
```

## Step 2: Introduce a Minimal Data Model 🧩

- Right now, our system handles _things_ — but it does not _know_ what those
  things are.
- Messages exist only as:
  - A string in a form.
  - A string in an HTTP request.
  - A string passed through functions.
- Follow state exists only as:
  - A boolean in React state.
  - A UI toggle.
- There is no _entity_.
- There is no identity.
- There is nothing to persist.
- Before a system can persist something, it must define what exists.

> **Persistence requires a model.**

- So the first step is to define what exists — inside this route.

### Step 2a: Decide what exists in this system

- Right now, the system has two real concepts:
  - A **Message**.
  - A **Profile**.
- A Message is something the user creates.
- A Profile is something the user can interact with and change (follow/unfollow).
- These are not UI concepts.
- They are system concepts.
- So we model both.

## Step 3: Introduce a Minimal Persistent Store 💾

- We have now defined what exists.
- But nothing is remembered yet.
- The system still forgets everything when the process restarts.
- Right now:
  - Messages live only in memory.
  - Profile follow state lives only in memory.
  - A reload destroys all truth.
- So the next step is to give the system **somewhere to remember**.

> **Persistence means storing truth outside the running process.**

- We will start with the smallest possible persistent store:
  a JSON file on disk.
- Not because it is scalable.
- Not because it is fast.
- But because it is:
  - Real
  - Visible
  - Honest
  - And teaches the right boundary.

<hr className='my-8' />

### Step 3a: Create a JSON store for the system

```ts
import fs from 'fs/promises'
import path from 'path'

const STORE_PATH = path.join(process.cwd(), 'app/data/store.json')

export type Store = {
  messages: {
    id: string
    text: string
    createdAt: number
  }[]
  profile: {
    id: string
    isFollowing: boolean
  }
}

export async function readStore(): Promise<Store> {
  const raw = await fs.readFile(STORE_PATH, 'utf-8')
  return JSON.parse(raw)
}

export async function writeStore(store: Store) {
  await fs.writeFile(STORE_PATH, JSON.stringify(store, null, 2))
}
```

```ts
export async function action({ request }: Route.ActionArgs) {
	const formData = await request.formData()
	const intent = formData.get('intent')

	if (intent === 'create') {
		const message = formData.get('message')

		if (typeof message !== 'string') {
			return { ok: false, error: 'A message must be a string.' }
		}

		const trimmedMessage = message.trim()

		if (trimmedMessage.length === 0) {
			return { ok: false, error: 'A message must not be empty.' }
		}

		const minLength = 3
		const maxLength = 200

		if (trimmedMessage.length < minLength) {
			return {
				ok: false,
				error: `A message must be at least ${minLength} characters.`,
			}
		}

		if (trimmedMessage.length > maxLength) {
			return {
				ok: false,
				error: `A message must be at most ${maxLength} characters.`,
			}
		}

		const store = await readStore()

		const newMessage = {
			id: randomUUID(),
			text: trimmedMessage,
			createdAt: Date.now(),
		}

		store.messages.push(newMessage)

		await writeStore(store)

		return { ok: true }
	}

	return { ok: false, error: 'Unsupported intent.' }
}
```

<hr className='my-8' />

- But updates must obey **the exact same message rules**:
  - A message must be a string.
  - A message must not be empty.
  - A message must respect length constraints.
- Duplicating that logic would be a mistake.

#### Define validation result types

- Validation has two possible outcomes:
  - Failure (with an error)
  - Success (with a guaranteed valid message)
- We model this explicitly using a **discriminated union**.
- Add the following **above the** `action` **function**:

```ts
type ValidationError = {
	ok: false
	error: string
}

type ValidationSuccess = {
	ok: true
	value: string
}

type ValidationResult = ValidationError | ValidationSuccess
```

- **Why this matters**
  - `ok` is the discriminator.
  - TypeScript can now **narrow types correctly**.
  - When `ok === true`, `value`:
    - Exists
    - Is a string
    - Is guaranteed to be valid
  - This removes string | undefined errors **by construction**.

#### Add the message validation helper

- Now we define validation once.
- It:
  - Accepts **unknown input**.
  - Enforces **all message rules**.
  - Returns a **typed result**.
- Add this function **below the types** and **above the** `action` **function**:

```ts
function validateMessageText(text: unknown): ValidationResult {
	if (typeof text !== 'string') {
		return { ok: false, error: 'A message must be a string.' }
	}

	const trimmedText = text.trim()

	if (trimmedText.length === 0) {
		return { ok: false, error: 'A message must not be empty.' }
	}

	const minLength = 3
	const maxLength = 200

	if (trimmedText.length < minLength) {
		return {
			ok: false,
			error: `A message must be at least ${minLength} characters.`,
		}
	}

	if (trimmedText.length > maxLength) {
		return {
			ok: false,
			error: `A message must be at most ${maxLength} characters.`,
		}
	}

	return { ok: true, value: trimmedText }
}
```

- **What this guarantees**
  - The function always returns **one canonical result**.
  - When `ok === true`, `value` is:
    - Trimmed
    - Valid
    - Safe to persist
  - This function now defines:
    > - **What a valid message is — system-wide**

```ts
	}

	await writeStore(store)

	return { ok: true }
}
```

#### Why this is the correct shape

- Delete requires exactly one thing: **identity**
- We:
  - Validate the ID
  - Remove the message from persistent storage
  - Persist the updated truth
- If nothing changed, we fail loudly
  > - Delete does not mutate data.
  > - Delete removes data.
- This keeps deletion:
  - Explicit
  - Honest
  - Server-authoritative

<hr className='my-8' />

### Step 7b: Add delete controls to the UI

- The server now understands how to delete a message.
- The remaining question is simple:
  > **How does the user tell the server which message to delete?**
- Answer:

  > - By submitting a form with:
  >   - `intent=delete` and **the message** `id`

- No abstractions. No tricks.

#### Where this change lives

- This code belongs **inside the message list**, specifically in
  the **non-editing state**.

You already have logic like this:

```tsx
{messages.slice(-3).map(message => {
	const isEditing = editingMessageId === message.id

	return (
		<li key={message.id}>
			{isEditing ? (
				// edit UI
			) : (
				// non-editing UI
			)}
		</li>
	)
})}
```

- We only change the **non-editing branch**.

#### Before (non-editing state)

```tsx
<>
	<span className='flex-1'>{message.text}</span>

	<button
		type='button'
		className='rounded-md border px-2 py-1 text-xs transition hover:bg-slate-100 hover:border-slate-400'
		onClick={() => {
			setEditingMessageId(message.id)
			setEditText(message.text)
		}}
	>
		Edit
	</button>
</>
```

#### After (non-editing state)

```tsx
<>
	<span className='flex-1'>{message.text}</span>

	<button
		type='button'
		className='rounded-md border px-2 py-1 text-xs transition hover:bg-slate-100 hover:border-slate-400'
		onClick={() => {
			setEditingMessageId(message.id)
			setEditText(message.text)
		}}
	>
		Edit
	</button>

	<Form method='post'>
		<input type='hidden' name='intent' value='delete' />
		<input type='hidden' name='id' value={message.id} />

		<button
			type='submit'
			className='rounded-md border px-2 py-1 text-xs text-red-600 transition hover:bg-red-50'
		>
			Delete
		</button>
	</Form>
</>
```

#### Why this is the _right_ shape

- **Edit**
  - Enters local draft state
  - Does not mutate truth yet
- **Delete**
  - Skips local state entirely
  - Submits an explicit server mutation
  - Removes persisted truth
  - Lets the loader re-render reality
- No conditionals.
- No UI cleanup logic.
- No guessing.

> - Intent goes to the server.
> - Truth changes.
> - UI follows.

- Triggers a loader re-run automatically
