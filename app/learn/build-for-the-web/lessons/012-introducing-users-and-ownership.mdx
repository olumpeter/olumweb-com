## Introduction 🧭

- In Lesson 10, something important happened.
- You introduced **persistent truth**.
- Messages and profile state stopped living in React and moved to the server.
- In Lesson 11, you proved something deeper:

  > Infrastructure can change without breaking the system.

- You replaced:
  - A JSON file
  - Manual file reads
  - Manual file writes
- With:
  - SQLite
  - Prisma
  - A real database
- And nothing above the persistence boundary changed.
- The system remained calm.
- But there is still a silent limitation.
- Right now:
  - All messages belong to **no one**
  - The profile belongs to **everyone**
- If two people opened this application at the same time:
  - They would see the same messages
  - They would share the same follow state
  - They would be editing the same profile
- In other words:

  > The system has persistence — but it does not yet have identity.

- A real system must answer a fundamental question:

  > Who does this data belong to?

## The Constraint 🧩

- We will not build authentication.
- We will simulate a user.
- The goal is **ownership**, not login.

## Mental Model 🧠

- Before:

```text
Message
Profile
```

- After

```text
User
  ⊦ Messages
  ⨽ Profile
```

> Identity creates boundaries.

## What You'll Build 🎯

- Messages belong to users
- Profiles belong to users
- Data becomes scoped
- UI remains unchanged

## Step 1: Starting Point 🏁

- Start from the end of Lesson 11.
- Your system currently includes:
  - React Router actions
  - Prisma persistence
  - SQLite database
  - CRUD for messages
  - Persistent profile state
- The database models currently look like this:

```prisma
model Message {
  id        String @id
  text      String
  createdAt Int
}

model Profile {
  id          String @id
  isFollowing Boolean
}
```

- Notice what is missing.

```text
User
```

- There is no concept of a user in the system.
- That means:
  - Messages do not belong to anyone
  - The profile does not belong to anyone
  - All data is effectively **global**
- This creates a limitation:

```text
Everyone shares the same data
```

- That works for a single-user system.
- But it breaks immediately in a multi-user system.

```text
The system can store data, but it cannot assign ownership.
```

- Before we can introduce ownership, we must first introduce identity.

## Step 2: Introduce Identity Into the Schema 👤

- Before ownership can exist, the system must define:

> Who owns data?

- Right now the schema defines:
  - `Message`
  - `Profile`
- But there is no concept of _User_
- We begin by introducing it.

### Step 2a: Add the User Model

- Open:

```text
prisma/schema.prisma
```

- Add:

```prisma
model User {
  id   String @id @default(cuid())
  name String
}
```

#### What this does

- Introduces identity into the system
- Users now exist — but nothing belongs to them yet

> Identity comes before ownership

#### Full schema after this step

- Your `schema.prisma` should now look like this:

```prisma
model Message {
  id        String @id
  text      String
  createdAt Int
}

model Profile {
  id          String @id
  isFollowing Boolean
}

model User {
  id   String @id @default(cuid())
  name String
}
```

#### What to notice

```text
Message
Profile
User
```

- All models exist
- But they are independent

> No ownership has been introduced yet

### Step 2b: Attach Messages to Users

- Now we define ownership of messages.
- Update the `Message` model:

```prisma
model Message {
  id        String @id
  text      String
  createdAt Int

  userId String
  user   User @relation(fields: [userId], references: [id])
}
```

- Update the `User` model:

```prisma
model User {
  id   String @id @default(cuid())
  name String

  messages Message[]
}
```

#### What this means

```text
User → Messages
```

- Each message belongs to a user
- A user can have many messages

> Ownership of messages is now established.

#### Understanding `userId` vs `user` 🔍

```prisma
userId String
user User @relation(fields: [userId], references: [id])
```

- `userId` → stored in the database
- `user` → defines the relationship

> Databases store identifiers. Prisma understands relationships.

#### Full schema after this step

```prisma
generator client {
  provider = "prisma-client"
  output   = "../app/generated/prisma"
}

datasource db {
  provider = "sqlite"
}

model Message {
  id        String @id
  text      String
  createdAt Int

  userId String
  user   User   @relation(fields: [userId], references: [id])
}

model Profile {
  id          String  @id
  isFollowing Boolean
}

model User {
  id   String @id @default(cuid())
  name String

  messages Message[]
}
```

### Step 2c: Attach Profiles to Users

- Messages now have ownership.
- But the profile is still global.

```text
That is incorrect.
```

- A profile must belong to exactly one user.
- Update the `Profile` model:

```prisma
model Profile {
  id          String @id @default(cuid())
  isFollowing Boolean

  userId String @unique
  user User @relation(fields: [userId], references: [id])
}
```

- Update the `User` model:

```prisma
model User {
  id   String @id @default(cuid())
  name String

  messages Message[]
  profile  Profile?
}
```

#### What this means

```text
User → Profile (1:1)
```

- Each user has one profile
- Each profile belongs to one user

#### Full schema after this step

```prisma
model Message {
  id        String @id
  text      String
  createdAt Int

  userId String
  user   User   @relation(fields: [userId], references: [id])
}

model Profile {
  id          String  @id @default(cuid())
  isFollowing Boolean

  userId String @unique
  user   User   @relation(fields: [userId], references: [id])
}

model User {
  id   String @id @default(cuid())
  name String

  messages Message[]
  profile  Profile?
}
```

## Step 2d: Verify the Full Schema

- At this point, ownership is fully encoded in the database.

```text
User
  ⊦ messages
  ∟ Profile
```

> The schema now understands ownership.

## Step 3: Apply the Database Migration 🗄️

- The schema has changed.
- The database must reflect those changes.

- Run the following command in your terminal:

```bash
npx prisma migrate dev --name add-user-ownership
```

- This will:
  - update the database
  - regenerate Prisma client

### Note: Regenerating the Prisma Client ⚠️

- After modifying `schema.prisma`, you may see a TypeScript error like this:

```ts
Property 'user' does not exist on type 'PrismaClient'
```

- This happens because Prisma generates TypeScript types from the schema.
- Until the Prisma client is regenerated, TypeScript still believes the schema
  looks like this:

```text
Message
Profile
```

- Running the migration command normally regenerates the client automatically:

```bash
npx prisma migrate dev --name add-user-ownership
```

- If you still see the error after running the migration, regenerate the Prisma
  client manually:

```bash
npx prisma generate
```

- This command rebuilds the generated Prisma client based on the current schema.
- After running it, TypeScript will correctly recognize the new model:

```ts
prisma.user.findUnique()
prisma.user.create()
```

- If the error still appears in your editor, restart the TypeScript server:

```text
VS Code → Command Palette → "TypeScript: Restart TS Server"
```

- Once the Prisma client is regenerated, the error should disappear.

## Step 4: Simulate a Logged-In User 👤

> Who is currently using the system?

- In a real application:
  - The server determines this per request
  - Identity comes from authentication and sessions
- We have not built that yet.

> So we simulate a logged-in user.

### Step 4a: Define Identity

- Open:

```ts
app/data/store.server.ts
```

- Add:

```ts
const USER_ID = 'user-1'
```

#### What this represents

- This constant represents:

```text
The current user
```

- Every request behaves as if it comes from this user.

#### Important

- This is not real authentication
- It is a controlled simplification

> Lesson 12 handles ownership
> Lesson 13 will handle real identity per request

### Step 4b: Ensure the User Exists

- The schema now requires that all data belongs to a user.

> The current user must exist in the database.

- Open the `readStore` function in:

```text
app/data/store.server.ts
```

- Locate the start of the function:

```ts
export async function readStore(): Promise<Store> {
```

- Add the following code **at the very top of the function, before any queries**:

```ts
let user = await prisma.user.findUnique({
  where: { id: USER_ID },
})

if (!user) {
  user = await prisma.user.create({
    data: { name: 'Peter Olum' },
  })
}
```

#### Why this is necessary

- You cannot assign ownership to a user that does not exist

```text
USER_ID → must map to a real database record
```

#### Important detail

- We do not assign id
- Prisma generates it automatically

> Identity is created by the system, not manually

### Step 4c: Ensure the Profile Exists

- The schema now enforces:

```text
User → Profile
```

- That means:

> Each user must have a profile

- Still inside the `readStore` function in:

```text
app/data/store.server.ts
```

- Locate the existing profile logic (the code that reads or creates the profile)
- Replace it with the following:

```ts
let profile = await prisma.profile.findUnique({
  where: { userId: USER_ID },
})

if (!profile) {
  profile = await prisma.profile.create({
    data: { userId: USER_ID, isFollowing: false },
  })
}
```

#### What changed

- Before:
  - Profile was global
- Now:
  - Profile belongs to a user

```text
userId → ownership
```

#### Step 4d: Scope Queries to the Current User

- Ownership exists in the database.
- Now the application must respect that ownership when reading data.
- Update the query:

- Still inside the `readStore` function in:

```text
app/data/store.server.ts
```

- Locate the section where messages are being fetched.
- Replace that query with:

```ts
const messages = await prisma.message.findMany({
  where: { userId: USER_ID },
  orderBy: { createdAt: 'asc' },
})
```

#### Before vs After

```text
Before: all messages
After: only this user's messages
```

> Ownership must be enforced when reading

### Step 4e: Record Ownership When Creating Messages

- Reading is now scoped to the user.
- Writing must also record ownership.
- Open:

```text
app/routes/home.tsx
```

- Locate your `action` function (the function that handles form submissions):

```ts
export async function action({ request }: Route.ActionArgs) {
```

- At the very top of this function, add:

```ts
const USER_ID = 'user-1'
```

#### Why define it again?

- Each file runs independently
- Variables are not shared automatically

> In real systems, this comes from the request

- Still inside the `action` function in:

```text
app/routes/home.tsx
```

- Locate the existing message creation code:

```ts
await prisma.message.create({
  data: {
    ...newMessage,
  },
})
```

- Update it by adding `userId` to the `data` object:

```ts
await prisma.message.create({
  data: {
    ...newMessage,
    userId: USER_ID,
  },
})
```

#### What this does:

- Every message now records:

```text
Who created it
```

> Ownership must exist at creation time

#### Step 4f: Important Clarification

- You may notice something:
  - The database generates user IDs automatically:

```prisma
id String @id @default(cuid())
```

- But we are using:

```ts
const USER_ID = 'user-1'
```

#### Why this still works

- We are simulating a stable identity
- The system always refers to the same user
- In a real system:

```ts
Request → Session → userId
```

- That will replace this constant in the next lesson

> `USER_ID` is a temporary stand-in for real authentication

### Step 4g: Mental Model Check

- The system now behaves like this:

```text
Request (simulated)
    ↓
USER_ID
    ↓
Database
    ↓
User → Messages + Profile
```

- Even without authentication:

> The system behaves as if a user is logged in

### Step 4 Summary

- Ownership is now enforced at every level:
  - Schema
  - Queries
  - Writes
- The UI has not changed
- But the system has fundamentally changed

> The system now understands who owns the data
